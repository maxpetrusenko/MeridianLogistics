PERMISSIONS

MODEL

Access is scoped by office and role. A broker can only see shipments, carriers, shippers, and
invoices where they are the assigned broker within their office. An office_manager can see all
records within their office. A vp can see all offices. The agent must enforce these boundaries on
every query it generates. A Dallas broker asking “show me all shipments” must only see Dallas
shipments assigned to them, never Atlanta’s data.

THE

PROVOCATION

“

Every screen in FreightView is just a different shaped window into the same database. What if we
replaced all of them with one interface: a chat?
CEO, Meridian Logistics

The VP of Engineering has 90 days and four engineers to build a proof-of-concept. The goal for this
phase is a working agent that handles read queries and single-step bookings for the Memphis
office, with the architecture designed to extend to multi-office, multi-step workflows in a later
phase. An initial prototype that passes the full schema and a broker question directly to an LLM
generates valid SQL only 62% of the time. Common failures include hallucinated column names,
incorrect joins, and unbounded queries returning millions of rows.

ASSIGNMENT

You are the VP of Engineering. Design the architecture for a conversational agent

that replaces FreightView’s six-module CRUD interface with a single chat. Your
architecture should be scoped to the 90-day PoC (read queries + single-step
bookings, Memphis office), but designed so the five areas below can extend to full
multi-office, multi-step production use.
Your proposal must address five areas:
Data Querying: How does the agent translate natural language into safe, correct SQL
against this schema? What prevents destructive queries, sensitive column exposure, and
unbounded result sets?
Multi-step Workflows: A broker says “find me the cheapest carrier and book it.” That’s
search → quote → confirm → book, with multiple steps involving branching, failures, and

human approval. How do you orchestrate that? What happens when a quote expires
mid-conversation?
Action Execution: The agent needs to call FreightView’s 34 API endpoints, some read-

only and some with financial consequences. How does the agent discover and invoke
these actions? How do you enforce broker confirmation before writes execute?

Security & Evaluation: How do you defend against prompt injection, cross-office data
leakage, and unauthorized action escalation? A Dallas broker should never see Atlanta’s

data, even if they ask cleverly. How do you measure accuracy and build an evaluation
harness that evolves as the agent’s capabilities expand?
Rich Responses: Freight data is useless as plain text. Brokers need sortable tables,
status timelines, and interactive confirmation cards, all rendered inline in chat. What is
the contract between agent output and the frontend? How do interactive elements

trigger further actions?

Your deliverable should include:
A system architecture diagram showing every component (LLM, API gateway, query
engine, tool router, permission layer, frontend) and the data flows between them.
Interface contracts for at least two boundaries: agent-to-database (e.g., what the
generated SQL or query plan looks like before execution) and agent-to-frontend (e.g.,
the structured response format the chat UI renders). Use the reference payloads below

as a starting point.
A security model specifying where permission checks run, what gets filtered, and how
you prevent bypass.
An evaluation plan with concrete metrics (accuracy, latency, safety) and how you would
build your initial test suite from the six example queries above.
A scope table that separates what ships in the 90-day PoC from what is deferred to
later phases, with brief rationale for each deferral.
