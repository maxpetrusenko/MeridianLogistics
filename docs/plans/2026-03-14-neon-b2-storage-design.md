# Neon B2 Storage Design

## Goal

Adopt the cheapest production-shaped data stack for the Meridian Memphis PoC: Neon Postgres for relational state and Backblaze B2 for generated artifacts.

## Recommendation

Use:

- Neon Postgres for queryable operational data, job state, and artifact metadata
- Backblaze B2 for generated PDFs, exports, replay fixtures, and other large blobs
- Railway for backend compute
- Netlify for frontend delivery

This matches the current relational source model, keeps blob bytes out of Postgres, and minimizes recurring storage cost.

## Why This Fits Meridian

The authoritative source set describes a relational system with office-scoped permissions, multi-table queries, ranking, aggregation, and booking confirmation flows. That shape fits Postgres directly. The same source set also implies generated artifacts and document-like outputs that do not need relational storage.

The design therefore splits storage by access pattern:

- relational and transactional data in Postgres
- immutable or large generated bytes in object storage

## Data Placement

### Neon Postgres

Store:

- PoC relational tables already in `db/schema.sql`
- future `generation_jobs`
- future `documents_manifest`
- future audit or replay metadata rows
- object keys, checksums, mime types, sizes, and status fields

Do not store:

- generated PDFs
- full replay payload blobs
- large model transcripts or export files

### Backblaze B2

Store:

- generated PDFs
- large JSON exports
- replay fixtures
- redacted logs or artifact bundles
- optional source-document copies

Do not store:

- queryable operational state
- permission context
- confirmation state

## Runtime Flow

1. API receives a generation or export request.
2. App creates a `generation_job` row in Postgres.
3. Worker generates the artifact on ephemeral disk only long enough to upload.
4. App uploads bytes to B2 under a deterministic object key.
5. App writes object metadata back to Postgres.
6. Frontend fetches job status and artifact metadata from the API.
7. API returns a signed download URL or proxied download response.

## Agent Compatibility

Agents work against API and database abstractions, not directly against local disk.

- query and orchestration agents read or write relational state through Postgres-backed app logic
- artifact-producing agents write bytes through the storage adapter
- frontend and eval flows consume metadata from Postgres and artifact bytes from B2

Because B2 exposes an S3-compatible API, the storage adapter can stay provider-shaped rather than vendor-shaped.

## Secrets

Required backend secrets:

- `MERIDIAN_DATABASE_URL`
- `MERIDIAN_DIRECT_DATABASE_URL`
- `MERIDIAN_B2_ENDPOINT`
- `MERIDIAN_B2_BUCKET_NAME`
- `MERIDIAN_B2_ACCESS_KEY_ID`
- `MERIDIAN_B2_SECRET_ACCESS_KEY`
- `OPENAI_API_KEY`

Likely additional app secrets:

- auth signing secret or JWT validation secret
- signed-download secret if the app signs its own artifact URLs

Frontend should only need public API configuration.

## Cost Shape

Neon provides the relational tier. B2 is the cheap blob tier.

Expected recurring pattern:

- Postgres cost dominated by compute plan and relational storage
- B2 cost dominated by stored GB, usually tiny for PoC artifact volume

Keep PDFs and exports in B2 so Postgres remains small, cheap, and fast.

## Implementation Scope

For this repo, the implementation should:

- add config support for Neon-style direct and app URLs
- add config support for B2 bucket credentials
- add a storage context and adapter boundary
- add manifest and job tables to the schema
- document env vars and local-dev fallback behavior

## Deferred

- real upload worker execution
- download signing implementation details beyond adapter seam
- lifecycle retention jobs
- production migration wiring
